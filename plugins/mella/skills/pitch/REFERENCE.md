# Pitch — Analysis Reference

## What to Analyse

- **Architecture & patterns** — How is the code structured? What abstractions exist?
- **Public API surface** — What does the project expose? Where are the rough edges?
- **Test coverage & quality** — What's well-tested? Where are the gaps?
- **Dependencies** — Heavy, outdated, or surprising deps?
- **Config & build** — How is it configured, built, published?
- **README & docs** — What does the project promise? Where do docs fall short?
- **Pain points** — Where is the code complex, repetitive, or fragile?
- **DX friction** — Where would a new contributor struggle? Where do power users hit walls?

## Analysis Mindset

Think like a senior engineer and product thinker combined. Look for:

- **Leverage points** — small changes that unlock disproportionate value
- **Missing abstractions** — patterns that repeat but aren't captured
- **Capability gaps** — things users probably expect but can't do yet
- **DX wins** — better errors, smarter defaults, less boilerplate
- **Integration opportunities** — natural connections to other tools/services
- **Underexploited strengths** — things the project does well but doesn't lean into
